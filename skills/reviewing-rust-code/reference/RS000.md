# RS000: missing-invariant-assertion

## Problem

A function relies on invariants that the type system does not capture,
but does not assert them.
Silently proceeding with a violated invariant can produce wrong results
or undefined behavior (in `unsafe` code),
while a `debug_assert!` would catch the violation immediately during testing.

Rust's type system captures many invariants that C programmers must assert manually
(nullability, ownership, bounds),
so blanket assertion-count targets do not translate directly.
The rule applies most strongly to invariants the type system *cannot* express:
ordering assumptions, relationships between fields,
and preconditions on `unsafe` operations.

## Pros / Cons / Decision

**Pros of asserting invariants:**
- Catches violations immediately during testing instead of producing silent corruption.
- Documents the assumption in executable form.
- `debug_assert!` has zero cost in release builds.

**Cons:**
- Assertions on large data (e.g., "slice is sorted") can be expensive even in debug builds.
- Over-asserting trivially-correct code adds noise.

**Decision:**
Add `debug_assert!` for non-trivial invariants
that the type system cannot capture,
especially preconditions on `unsafe` code and algorithm-specific assumptions.

## Example

### Bad
```rust
/// Merges two sorted slices into a single sorted Vec.
fn merge_sorted(a: &[i32], b: &[i32]) -> Vec<i32> {
    // Assumes a and b are sorted, but never checks.
    let mut result = Vec::with_capacity(a.len() + b.len());
    // ... merge logic ...
    result
}
```

### Good
```rust
fn merge_sorted(a: &[i32], b: &[i32]) -> Vec<i32> {
    debug_assert!(a.windows(2).all(|w| w[0] <= w[1]), "a must be sorted");
    debug_assert!(b.windows(2).all(|w| w[0] <= w[1]), "b must be sorted");
    let mut result = Vec::with_capacity(a.len() + b.len());
    // ... merge logic ...
    debug_assert!(result.windows(2).all(|w| w[0] <= w[1]), "result must be sorted");
    result
}
```

## When to flag

- A function's doc comment or name implies preconditions
  (e.g., "sorted", "non-empty", "normalized")
  that are not checked, even in debug mode.
- An `unsafe` block relies on invariants beyond what the SAFETY comment describes,
  with no `debug_assert!` to validate them.
- A complex algorithm relies on a loop invariant or postcondition
  that could cheaply be verified with `debug_assert!`.

## When NOT to flag

- The invariant is enforced by the type system
  (e.g., `NonZeroUsize`, a validated newtype).
- Adding a `debug_assert!` would require O(n) or worse work,
  making it impractical even in debug builds.
- Simple functions where the logic is obviously correct by inspection.
- The function already uses `assert!` or returns `Result` to validate the condition.

[^1]: Inspired by [Power of 10 Rule 5](https://en.wikipedia.org/wiki/The_Power_of_10:_Rules_for_Developing_Safety-Critical_Code): *"The code's assertion density should average to minimally two assertions per function."*
