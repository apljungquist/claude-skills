# D045: missing-pub-doc-comment

**Level:** default

## Problem

A public item (function, method, type, trait, constant) has no `///` doc comment.
Callers who cannot see the implementation must read source code to understand the contract, defeating the purpose of an API boundary.
Undocumented public APIs are harder to use correctly and harder to maintain because there is no stated contract to validate against.

## Pros / Cons / Decision

**Pros of requiring doc comments on public items:**
- Callers can understand the contract without reading the implementation.
- IDEs surface doc comments on hover.

**Cons:**
- Adds writing overhead for obvious accessor methods.

**Decision:**
Require doc comments on all public items.
Exempt obvious accessors and trait implementations where the trait itself is documented.

## Example

### Bad
```rust
pub fn compress(data: &[u8], algorithm: Algorithm) -> Result<Vec<u8>> {
    // No doc comment — callers must read the source to understand
    // size limits, error conditions, and algorithm behavior.
    // ...
}

pub struct ConnectionPool {
    // No doc comment — what is the lifecycle? Thread-safe?
    // ...
}
```

### Good
```rust
/// Compresses the input data using the specified algorithm.
///
/// Returns the compressed bytes, or an error if the input exceeds
/// the maximum supported size (4 GiB).
pub fn compress(data: &[u8], algorithm: Algorithm) -> Result<Vec<u8>> {
    // ...
}

/// A thread-safe pool of reusable database connections.
///
/// Connections are lazily created up to `max_size` and returned to the
/// pool when dropped. If all connections are in use, callers block
/// until one becomes available.
pub struct ConnectionPool {
    // ...
}
```

## When to flag

- Any `pub` function, method, struct, enum, trait, or constant without a `///` doc comment.
- `pub` items with only `//` comments (regular comments, not doc comments).
- Doc comments that are empty or contain only the item name (e.g., `/// ConnectionPool`).

## When NOT to flag

- Private items (`fn`, `pub(crate)`, `pub(super)`) — these are implementation details and are exempt.
- Trait implementations where the trait itself is documented and the implementation does not deviate from the documented contract.
- Simple re-exports (`pub use`) where the original item is documented.
- Items in `#[cfg(test)]` modules.
- `pub` items in binary crates (`main.rs`) that are not part of a library API.
- Obvious accessor methods like `fn id(&self) -> &Id` where the name and signature fully describe the behavior.

**Related rules:** D010 `doc-contradicts-code`, N038 `name-scope-mismatch`.

[^1]: Inspired by [Linux kernel coding style §8](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#commenting): *"Comments at the head of functions should describe what they do, not how they do it."* The kernel guide requires kernel-doc formatted comments for API functions. In Rust, `///` doc comments serve this role and are rendered by `cargo doc`, surfaced by IDEs on hover, and published to docs.rs.
