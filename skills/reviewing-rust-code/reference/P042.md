# P042: excessive-inline

**Level:** default

## Problem

`#[inline]` or `#[inline(always)]` is applied to functions that are too large for the compiler to benefit from inlining, or applied without benchmarking evidence.
The Rust compiler and LLVM already make good inlining decisions; overriding them without measurement is premature optimization that increases compile times and binary size.

## Pros / Cons / Decision

**Pros of letting the compiler decide inlining:**
- LLVM makes good inlining decisions based on actual call patterns.
- Avoids binary size bloat from forced inlining.

**Cons:**
- Cross-crate inlining requires `#[inline]` for generic functions.
- Trivial accessors benefit from an explicit hint.

**Decision:**
Use `#[inline]` only for trivial accessors and cross-crate generic functions.
`#[inline(always)]` requires benchmark evidence.

## Example

### Bad
```rust
// 30-line function forced inline without evidence
#[inline(always)]
fn process_batch(items: &[Item], config: &Config) -> Result<Vec<Output>> {
    let mut results = Vec::with_capacity(items.len());
    for item in items {
        let validated = validate(item, config)?;
        let transformed = transform(&validated)?;
        results.push(Output::from(transformed));
    }
    Ok(results)
}
```

### Good
```rust
#[inline]
fn is_empty(&self) -> bool {
    self.len == 0
}

// Large function — let the compiler decide
fn process_batch(items: &[Item], config: &Config) -> Result<Vec<Output>> {
    // ...
}
```

## When to flag

- `#[inline(always)]` on a function longer than 3–5 lines without a comment citing benchmark results.
- `#[inline]` on functions that are not trivial accessors, small wrappers, or hot-path functions with compile-time-constant parameters.
- Blanket `#[inline]` on all methods of an impl block.

## When NOT to flag

- `#[inline]` on trivial accessors and predicates (1–3 lines).
- `#[inline]` on generic functions in library crates, where inlining across crate boundaries requires the attribute (the compiler cannot inline cross-crate without it).
- `#[inline(always)]` with an accompanying comment citing benchmark data.
- Functions in `#[no_std]` or embedded contexts where binary size and performance tradeoffs are well-understood.

[^1]: Inspired by [Linux kernel coding style §15](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#the-inline-disease): *"A reasonable rule of thumb is to not put inline at functions that have more than 3 lines of code in them."* The kernel guide calls excessive inlining "the inline disease" — it increases binary size, pollutes the instruction cache, and can make code slower, not faster.
