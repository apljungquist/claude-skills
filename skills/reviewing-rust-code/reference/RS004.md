# RS004: comment-restates-code

## Problem

A comment inside a function body restates what the code already says in English, adding no information.
When the code changes but the comment doesn't, the comment becomes a lie — which is worse than no comment at all.
Restating comments also add visual noise that makes the code harder to scan.

## Pros / Cons / Decision

**Pros of removing restating comments:**
- Reduces maintenance burden when code changes but comments don't.
- Improves signal-to-noise ratio in the code.

**Cons:**
- Some developers prefer comments on every block for orientation.

**Decision:**
Comments should explain *why*, not *what*.
Flag comments that merely translate code to English.

## Example

### Bad
```rust
// Loop from 0 to MAX_RETRIES
for attempt in 0..MAX_RETRIES {
    // Send the request
    match client.send(&request).await {
        // If ok, return the response
        Ok(response) => return Ok(response),
        // If error and not last attempt, sleep
        Err(_) if attempt < MAX_RETRIES - 1 => {
            sleep(Duration::from_millis(100 << attempt)).await;
        }
        // Otherwise return the error
        Err(e) => return Err(e.into()),
    }
}
```

### Good
```rust
// Retry with exponential backoff because the upstream service
// rate-limits clients that reconnect too aggressively.
for attempt in 0..MAX_RETRIES {
    match client.send(&request).await {
        Ok(response) => return Ok(response),
        Err(_) if attempt < MAX_RETRIES - 1 => {
            sleep(Duration::from_millis(100 << attempt)).await;
        }
        Err(e) => return Err(e.into()),
    }
}
```

## When to flag

- A comment that translates a single line of code into English (e.g., `// increment counter` above `counter += 1`).
- Per-arm comments in a `match` that simply describe the pattern (e.g., `// If Ok` above the `Ok` arm).
- Comments that describe control flow that is already visible from the code structure (e.g., `// return early if empty` above `if items.is_empty() { return Ok(()); }`).

## When NOT to flag

- Comments that explain *why* a non-obvious approach was chosen (e.g., why a specific algorithm, why a workaround for a known issue).
- Comments that provide domain context not visible in the code (e.g., a reference to a specification or protocol section).
- `// SAFETY:` comments on `unsafe` blocks — these are required and explain invariants, not code mechanics.
- Section-separator comments in long functions that help orientation (though long functions should be split — see N032 `excessive-function-length`).
- Comments that describe the intent of a complex expression (e.g., a non-obvious bitwise operation).

**Related rules:** D010 `doc-contradicts-code`, D011 `stale-safety-comment`.

[^1]: Inspired by [Linux kernel coding style §8](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#commenting): *"Comments are good, but there is also a danger of over-commenting. NEVER try to explain HOW your code works in a comment: it's much better to write the code so that the working is obvious."* The kernel guide emphasizes: tell WHAT your code does, not HOW.
