# L030: unbounded-recursion

**Level:** default

## Problem

A recursive function has no depth limit
or the base case is not obviously reachable for all inputs.
Rust does not optimize tail calls,
so deep or unbounded recursion causes a stack overflow
that aborts the process with no opportunity for recovery.

## Pros / Cons / Decision

**Pros of bounding or eliminating recursion:**
- Prevents stack overflow on adversarial or malformed input.
- Makes resource usage predictable and auditable.

**Cons:**
- Iterative rewrites can be harder to read for naturally recursive structures (trees, grammars).
- Adding a depth parameter adds boilerplate to the API.

**Decision:**
Require a depth limit or iterative rewrite
for recursive functions that process external input.
Recursion on compile-time-bounded structures is acceptable.

## Example

### Bad
```rust
fn flatten(value: &Value) -> Vec<String> {
    match value {
        Value::Leaf(s) => vec![s.clone()],
        Value::Node(children) => children
            .iter()
            .flat_map(flatten)  // Adversarial input can nest arbitrarily deep
            .collect(),
    }
}
```

### Good
```rust
fn flatten(value: &Value) -> Result<Vec<String>, Error> {
    flatten_inner(value, 0)
}

fn flatten_inner(value: &Value, depth: usize) -> Result<Vec<String>, Error> {
    if depth > MAX_DEPTH {
        return Err(Error::TooDeep);
    }
    match value {
        Value::Leaf(s) => Ok(vec![s.clone()]),
        Value::Node(children) => {
            let mut out = Vec::new();
            for child in children {
                out.extend(flatten_inner(child, depth + 1)?);
            }
            Ok(out)
        }
    }
}
```

Alternatively, convert to an iterative approach with an explicit stack:
```rust
fn flatten(value: &Value) -> Vec<String> {
    let mut result = Vec::new();
    let mut stack = vec![value];
    while let Some(current) = stack.pop() {
        match current {
            Value::Leaf(s) => result.push(s.clone()),
            Value::Node(children) => stack.extend(children.iter()),
        }
    }
    result
}
```

## When to flag

- Recursive function that processes external input
  (parsed data, user-provided trees, deserialized structures) without a depth limit.
- Indirect recursion (A calls B, B calls A)
  where the cycle is not immediately obvious.
- Recursive function where the base case depends on a property of the input
  that is not validated (e.g., assuming a graph is acyclic).

## When NOT to flag

- Recursion over structures with compile-time bounded depth
  (e.g., a fixed-depth type hierarchy).
- Recursion with an explicit depth counter or fuel parameter that is checked.
- Recursion in code that only processes trusted,
  internally-generated data of known bounded depth.
- Iterative rewrites that would make the code substantially harder to understand,
  where the input is bounded by other means.

[^1]: Inspired by [Power of 10 Rule 1](https://en.wikipedia.org/wiki/The_Power_of_10:_Rules_for_Developing_Safety-Critical_Code): *"Do not use direct or indirect recursion."*
