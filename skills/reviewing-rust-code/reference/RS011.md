# RS011: println-for-diagnostics

## Problem

`println!` or `eprintln!` is used for diagnostic or operational output where a logging facade (`log`, `tracing`) would be more appropriate.
Direct printing bypasses the application's logging configuration, cannot be filtered by level or module, is invisible to structured-log pipelines, and cannot be silenced without removing the code.

The Python logging documentation puts it plainly: *use `print()` for console output of a normal command-line script; use logging for reporting events that occur during normal operation.*
The same principle holds in Rust — `println!` is for user-facing CLI output; `log::info!` and friends are for operational diagnostics.

## Pros / Cons / Decision

**Pros of using a logging facade:**
- Output is filterable by level and module.
- Integrates with structured log pipelines.

**Cons:**
- Adds a dependency on `log` or `tracing`.
- CLI tools that print user-facing output legitimately use `println!`.

**Decision:**
Use `log` or `tracing` for diagnostic output.
Reserve `println!`/`eprintln!` for user-facing CLI output.

## Example

### Bad
```rust
// In a library crate
pub fn sync_data(source: &Url) -> Result<Stats, SyncError> {
    eprintln!("Starting sync from {source}");  // Bad: Not filterable, pollutes stderr
    let data = fetch(source)?;
    eprintln!("Fetched {} items", data.len());
    let stats = apply(data)?;
    eprintln!("Sync complete: {stats:?}");
    Ok(stats)
}

// In an HTTP handler
async fn handle_request(req: Request) -> Response {
    println!("Got request: {}", req.uri());  // Bad: Interleaves with structured logs
    // ...
}
```

### Good
```rust
use tracing::{info, debug};

pub fn sync_data(source: &Url) -> Result<Stats, SyncError> {
    info!(%source, "starting sync");
    let data = fetch(source)?;
    debug!(count = data.len(), "fetched items");
    let stats = apply(data)?;
    info!(?stats, "sync complete");
    Ok(stats)
}

async fn handle_request(req: Request) -> Response {
    debug!(uri = %req.uri(), "handling request");
    // ...
}
```

## When to flag

- `println!` or `eprintln!` used to report operational events (start, progress, completion, errors) in library crates.
- `println!` or `eprintln!` used for diagnostic output in server/daemon code where structured logging is already in use.
- `dbg!()` left in non-test code.

## When NOT to flag

- Binary crates that intentionally use `println!` for user-facing CLI output (e.g., query results, formatted reports, `--help` text).
- `eprintln!` in early startup before logging is initialized (e.g., printing a fatal error when the config file can't be loaded).
- `eprintln!` in panic hooks or signal handlers where the logging system may be unavailable.
- Code that does not depend on a logging facade and explicitly chooses to stay dependency-free.
- `dbg!()` in test code.
