# N040: multiple-responsibilities

**Level:** default

## Problem

A function performs multiple conceptually distinct operations.
If the function's purpose can be described using the word "and", it is doing more than one thing.
When either responsibility changes, the entire function must be re-examined, increasing the risk of regressions in unrelated behavior.
Multi-responsibility functions are also harder to name, test, and reuse.

## Pros / Cons / Decision

**Pros of single-responsibility functions:**
- Each function is independently testable.
- Changes to one responsibility don't risk regressions in the other.

**Cons:**
- Splitting tightly-coupled logic can scatter context across functions.
- Orchestration functions that compose steps are intentionally multi-step.

**Decision:**
Flag functions with clearly separable responsibilities.
Orchestration functions that compose smaller functions are acceptable.

## Example

### Bad
```rust
fn validate_and_store_user(input: &str, db: &Database) -> Result<UserId> {
    // Validation logic
    let email = input.parse::<Email>()
        .map_err(|_| Error::InvalidEmail)?;
    if db.email_exists(&email)? {
        return Err(Error::DuplicateEmail);
    }

    // Persistence logic (different responsibility)
    let id = db.insert_user(&email)?;
    db.send_welcome_email(&email)?;
    Ok(id)
}
```

### Good
```rust
fn validate_new_user(input: &str, db: &Database) -> Result<Email> {
    let email = input.parse::<Email>()
        .map_err(|_| Error::InvalidEmail)?;
    if db.email_exists(&email)? {
        return Err(Error::DuplicateEmail);
    }
    Ok(email)
}

fn store_user(email: &Email, db: &Database) -> Result<UserId> {
    let id = db.insert_user(email)?;
    db.send_welcome_email(email)?;
    Ok(id)
}
```

## When to flag

- Function name contains "and" or "or" describing distinct operations (e.g., `parse_and_validate`, `fetch_or_create`).
- Function has clearly separable phases — e.g., validation followed by persistence, computation followed by I/O, parsing followed by transformation — that could be independent functions.
- A function that could be meaningfully tested as two separate units but can only be tested as one.

## When NOT to flag

- Orchestration functions that compose smaller functions into a workflow — these are intentionally multi-step (e.g., `process_order` that calls `validate`, `charge`, `confirm`).
- Functions where splitting would require passing many intermediate values, making the decomposition harder to follow than the monolith.
- Trait implementations where the trait contract requires multiple operations in a single method.
- Short functions (under ~15 lines) where the "two responsibilities" are tightly coupled and trivial.

**Related rules:** N032 `excessive-function-length`.

[^1]: Inspired by [Linux kernel coding style §6](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#functions): *"Functions should be short and sweet, and do just one thing. They should fit on one or two screenfuls of text, and do one thing and do that well."*
