# A050: invariant-leaking-module-boundary

**Level:** default

## Problem

A type has invariants (e.g., sorted, non-empty, normalized)
but its module boundary allows external code to construct or mutate instances that violate those invariants.
`pub` fields, public tuple struct constructors, or missing validation in `new()`
break the encapsulation that makes the invariant trustworthy.
Once external code can create an invalid instance,
every function that receives the type must either re-validate or silently trust the caller —
the invariant is no longer enforced by the module.

## Pros / Cons / Decision

**Pros of sealing invariants at the module boundary:**
- Functions that accept the type can trust the invariant without re-checking.
- Invalid states are unrepresentable outside the module.

**Cons:**
- Private fields require accessor methods, adding boilerplate.
- Plain data types (DTOs) with no invariants do not benefit from encapsulation.

**Decision:**
Keep fields private and validate in constructors for types with invariants.
Use `pub` fields freely on plain data types where all field values are valid.

## Example

### Bad
```rust
/// A percentage value between 0 and 100.
pub struct Percentage {
    pub value: f64, // External code can set this to 200.0
}

impl Percentage {
    pub fn new(value: f64) -> Self {
        // No validation — any f64 is accepted
        Self { value }
    }
}
```

### Good
```rust
/// A percentage value between 0 and 100.
pub struct Percentage {
    value: f64, // Private — only this module can set it
}

impl Percentage {
    pub fn new(value: f64) -> Result<Self, Error> {
        if !(0.0..=100.0).contains(&value) {
            return Err(Error::OutOfRange);
        }
        Ok(Self { value })
    }

    pub fn value(&self) -> f64 {
        self.value
    }
}
```

## When to flag

- `pub` fields on a struct whose doc comment or name implies an invariant (e.g., "sorted", "non-empty", "normalized", "valid", "between X and Y").
- Public tuple struct (`pub struct Foo(pub T)`) where the inner value has constraints that the constructor does not enforce.
- `pub fn new()` that accepts arbitrary values for a type with documented invariants — the constructor should validate or return `Result`.
- Struct with a private invariant-enforcing constructor but a `#[derive(Default)]` that produces an invalid instance.

## When NOT to flag

- Plain data types or DTOs with no invariants — all field values are valid by definition.
- `pub(crate)` fields in small modules where the crate boundary is the enforcement point.
- Types where all possible field values satisfy the invariant (e.g., `pub struct Point { pub x: f64, pub y: f64 }`).
- Builder types whose fields are intentionally public for incremental construction before a final `build()` validates.
- Types in `#[cfg(test)]` modules where encapsulation is relaxed for testing.
