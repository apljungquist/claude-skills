# O031: log-level-mismatch

**Level:** default

## Problem

A log event uses a level that does not match the severity of the situation, making it harder to filter logs effectively and eroding operator trust in log output.
When errors are logged at `info` or routine progress is logged at `warn`, operators either drown in noise or miss real problems.

The following level guidelines are widely accepted across language ecosystems (Python, Java, Rust) and translate directly to the `log` and `tracing` crates:

| Level   | Use for |
|---------|---------|
| `error` | The operation failed and the caller or operator must act. Data loss, broken invariants, or unrecoverable failures. |
| `warn`  | Something unexpected happened but the system recovered or degraded gracefully. Merits investigation but not a page. |
| `info`  | Normal, notable milestones: server started, config loaded, job completed. Useful in production at default verbosity. |
| `debug` | Detailed diagnostic information for developers investigating issues. Not expected in production logs. |
| `trace` | Very fine-grained flow: entering/exiting functions, individual iterations. Typically only enabled during targeted debugging. |

## Pros / Cons / Decision

**Pros of correct log levels:**
- Operators can filter logs effectively.
- Alerts fire for real problems, not noise.

**Cons:**
- Level choice is sometimes borderline.

**Decision:**
Use levels consistently:
error for unrecoverable failures,
warn for recovered issues,
info for milestones,
debug/trace for diagnostics.

## Example

### Bad
```rust
// Bad: Routine success logged at warn — operators will learn to ignore warnings
log::warn!("Successfully processed {} records", count);

// Bad: Retryable failure logged at error — creates alert fatigue
log::error!("Timeout connecting to cache, retrying in 1s");

// Bad: Per-request detail at info — floods production logs
log::info!("Parsed request body: {body:?}");
```

### Good
```rust
// Good: Normal milestone at info
log::info!("Processed {count} records");

// Good: Transient, recovered failure at warn
log::warn!("Timeout connecting to cache, retrying in 1s");

// Good: Verbose diagnostic detail at debug or trace
log::debug!("Parsed request body: {body:?}");
```

## When to flag

- `error!` used for transient, retried, or recovered failures where the system continues normally.
- `warn!` used for routine, expected events (success messages, progress updates).
- `info!` used for per-item or per-request detail that would flood logs under normal load.
- `debug!` or `trace!` used for genuine errors that an operator should see.

## When NOT to flag

- Level choice is borderline (e.g., `warn` vs `error` for a failure with a fallback) — reasonable people disagree on edge cases.
- Log levels are configured dynamically or the code explicitly documents why a non-standard level was chosen.
- Test code — log levels in tests are often chosen for convenience.
