# S035: unnecessary-unsafe

**Level:** default

## Problem

An `unsafe` block is used where a safe alternative exists, or is broader than necessary.
Every `unsafe` block is a promise that the programmer has manually verified invariants the compiler cannot check.
Unnecessary `unsafe` expands the surface area a reviewer must audit and increases the risk of undefined behavior as the surrounding code evolves.

## Pros / Cons / Decision

**Pros of minimizing unsafe:**
- Less unsafe code means less surface area for undefined behavior.
- Easier for reviewers to audit the remaining unsafe blocks.

**Cons:**
- Performance-critical code may need `get_unchecked` to eliminate bounds checks.
- FFI requires unsafe by definition.

**Decision:**
Use safe alternatives when they exist.
When unsafe is necessary, scope the block as narrowly as possible.

## Example

### Bad
```rust
fn get_first(slice: &[u8]) -> Option<u8> {
    if slice.is_empty() {
        None
    } else {
        // Unnecessary — safe indexing or .first() does the same thing
        Some(unsafe { *slice.get_unchecked(0) })
    }
}
```

```rust
// Unsafe block is wider than needed
unsafe {
    let ptr = data.as_ptr();
    let len = data.len();       // This line doesn't need unsafe
    let header = parse(len);    // Neither does this
    std::ptr::read(ptr)         // Only this needs unsafe
}
```

### Good
```rust
fn get_first(slice: &[u8]) -> Option<u8> {
    slice.first().copied()
}
```

```rust
let ptr = data.as_ptr();
let len = data.len();
let header = parse(len);
// SAFETY: ptr is valid for reads of 1 element because data is non-empty (checked above).
let value = unsafe { std::ptr::read(ptr) };
```

## When to flag

- `unsafe` block where a safe standard library method achieves the same result (e.g., `get_unchecked` where indexing with bounds check is acceptable).
- `unsafe` block that spans many lines when only one operation requires it — the block should be narrowed.
- `unsafe` used for performance without evidence that the bounds check is a bottleneck.
- Raw pointer manipulation where references or slices would suffice.
- `unsafe impl Send/Sync` without a comment justifying why the type is safe to share across threads.

## When NOT to flag

- FFI calls (`extern "C"` functions) — `unsafe` is required.
- Performance-critical inner loops where profiling shows bounds checks are a measurable bottleneck, and the SAFETY comment explains the invariant.
- Operations that have no safe equivalent (e.g., `ptr::read`/`ptr::write` for manual memory management, `transmute` with documented layout guarantees).
- `unsafe` inside a safe abstraction that encapsulates the invariant (e.g., implementing a custom collection type), provided the SAFETY comments are accurate (see D011 `stale-safety-comment`).

[^1]: Inspired by [Power of 10 Rule 9](https://en.wikipedia.org/wiki/The_Power_of_10:_Rules_for_Developing_Safety-Critical_Code): *"Limit pointer use to a single dereference, and do not use function pointers."*
