# Rule: library-initializes-subscriber

**Severity:** warning

## Problem

A library crate initializes a global logger or tracing subscriber. This takes the decision away from the application, which should be the sole owner of logging configuration. Two libraries that both try to initialize a global subscriber will conflict, and the application loses control over output format, filtering, and destinations.

This is the Rust equivalent of the well-established Python practice: *libraries must never add handlers to their loggers — only a `NullHandler`. Configuration is the application developer's responsibility.* In Rust, the `log` and `tracing` facades already provide the equivalent of `NullHandler` by default — events are silently discarded if no subscriber is installed.

## Example

### Bad
```rust
// In a library crate's `lib.rs` or any module of a library
pub fn init() {
    env_logger::init();  // ❌ Library installs a global logger
    // ...
}

pub fn connect(addr: &str) -> Result<Connection, Error> {
    tracing_subscriber::fmt::init();  // ❌ Library installs a global subscriber
    // ...
}
```

### Good
```rust
// Library: just emit events through the facade — no initialization
use tracing::{info, warn};

pub fn connect(addr: &str) -> Result<Connection, Error> {
    info!(addr, "connecting");
    let conn = TcpStream::connect(addr).map_err(|e| {
        warn!(addr, %e, "connection failed");
        Error::Connect(e)
    })?;
    Ok(Connection(conn))
}

// Application (binary crate): owns subscriber setup
fn main() {
    tracing_subscriber::fmt()
        .with_env_filter("my_app=debug,my_lib=info")
        .init();
    // ...
}
```

## When to flag

- Calls to `env_logger::init()`, `env_logger::try_init()`, `env_logger::builder().init()` in a library crate.
- Calls to `tracing_subscriber::fmt::init()`, `tracing_subscriber::fmt().init()`, or `.set_global_default()` in a library crate.
- Calls to `log::set_logger()` or `log::set_max_level()` in a library crate.
- Any function in a library that installs a global logging or tracing subscriber as a side effect.

## When NOT to flag

- Binary crates (`main.rs`, CLI tools, servers) — these are the correct place to initialize subscribers.
- Test helper functions that set up a subscriber for test output (e.g., in `#[cfg(test)]` modules or integration test harnesses).
- Libraries that provide subscriber setup as an *opt-in* utility function clearly documented as application-level (e.g., a framework's `init_logging()` that the application explicitly calls from `main`).
- `try_init()` in test code using something like `tracing_subscriber::fmt().try_init().ok()` to avoid double-init panics — this is a common testing pattern.
