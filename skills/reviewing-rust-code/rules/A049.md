# A049: prefer-cfg-macro

**Level:** default

## Problem

`#[cfg(...)]` attributes are used on code blocks where `cfg!()` in an `if` expression would work.
With `#[cfg]`, the annotated code is completely invisible to the compiler when the condition is false.
If someone renames a function, changes a type, or alters a signature, the `#[cfg]`-gated code silently breaks — nobody discovers it until that feature is enabled.
With `cfg!()`, both branches are compiled and type-checked (the dead branch is eliminated by the optimizer).

## Pros / Cons / Decision

**Pros of `cfg!()` over `#[cfg]`:**
- Both branches are type-checked, catching breakage during refactoring.
- Dead branch is eliminated by the optimizer — no runtime cost.

**Cons:**
- `cfg!()` cannot be used when gated code depends on conditionally-available types.
- `#[cfg(test)] mod tests` is idiomatic.

**Decision:**
Prefer `cfg!()` when both branches compile.
Use `#[cfg]` only when gated code depends on conditionally-available types or imports.

## Example

### Bad
```rust
// If a refactor changes Level::Debug to Level::Trace,
// this code silently breaks when debug_assertions is off.
#[cfg(debug_assertions)]
fn log_level() -> Level {
    Level::Debug
}

#[cfg(not(debug_assertions))]
fn log_level() -> Level {
    Level::Info
}
```

### Good
```rust
fn log_level() -> Level {
    if cfg!(debug_assertions) {
        Level::Debug
    } else {
        Level::Info
    }
}
```

## When to flag

- Duplicate function definitions guarded by `#[cfg(X)]` and `#[cfg(not(X))]` where both branches use the same types and could be a single function with a `cfg!()` conditional.
- `#[cfg]` on a code block where all types, functions, and traits referenced in the block are available regardless of the feature flag.
- `#[cfg(test)]` on non-test items that could use `cfg!(test)` instead (note: `#[cfg(test)]` on `mod tests` is standard and should not be flagged).

## When NOT to flag

- `#[cfg]` where the gated code depends on types, traits, or imports that only exist under the feature flag — `cfg!()` cannot work because the dead branch would not compile.
- `#[cfg(test)] mod tests` — this is the standard Rust pattern for test modules.
- Platform-specific code that uses OS-specific APIs (e.g., `#[cfg(unix)]` calling `libc` functions).
- `#[cfg]` on `use` imports, `extern crate` declarations, or `mod` declarations where `cfg!()` is not syntactically valid.
- Feature-gated optional dependencies (`#[cfg(feature = "...")]` on `use` statements).

[^1]: Inspired by [Linux kernel coding style §21](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#conditional-compilation): *"Wherever possible, don't use preprocessor conditionals (#if, #ifdef) in .c files... Instead, prefer to use the IS_ENABLED() macro to convert the Kconfig symbols into C boolean expressions, and use it in a normal C conditional."* The principle: prefer constructs the compiler can type-check in both branches.
