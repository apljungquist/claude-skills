# Rule: too-many-locals

**Severity:** suggestion

Inspired by [Linux kernel coding style §6](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#functions): *"The number of local variables shouldn't exceed 5–10, or you're doing something wrong. Re-think the function, and split it into smaller pieces. A human brain can generally easily keep track of about 7 different things."*

## Problem

A function declares many local variable bindings, forcing the reader to hold too much state in their head to follow the logic. Functions with many locals are usually doing too many things (see N40 `multiple-responsibilities`) or computing intermediate values that belong in helper functions.

## Example

### Bad
```rust
fn generate_report(data: &Dataset) -> Report {
    let total = data.values().sum::<f64>();
    let count = data.len();
    let average = total / count as f64;
    let min = data.values().min_by(|a, b| a.partial_cmp(b).unwrap());
    let max = data.values().max_by(|a, b| a.partial_cmp(b).unwrap());
    let variance = data.values().map(|v| (v - average).powi(2)).sum::<f64>() / count as f64;
    let std_dev = variance.sqrt();
    let median = compute_median(data);
    let p90 = compute_percentile(data, 90);
    let p95 = compute_percentile(data, 95);
    let p99 = compute_percentile(data, 99);
    let skewness = compute_skewness(data, average, std_dev);
    // 12 locals — extract a helper for statistical measures
    Report { total, count, average, min, max, std_dev, median, p90, p95, p99, skewness, variance }
}
```

### Good
```rust
fn generate_report(data: &Dataset) -> Report {
    let basic = compute_basic_stats(data);
    let percentiles = compute_percentiles(data);
    let distribution = compute_distribution_stats(data, basic.average, basic.std_dev);
    Report::from_parts(basic, percentiles, distribution)
}
```

## When to flag

- A function has more than 10 `let` bindings (excluding `_` prefixed bindings and destructuring patterns that unpack a single value).
- The local variables can be grouped into coherent subsets that would make sense as a helper function or struct.

## When NOT to flag

- Destructuring a struct or tuple into named fields — this increases the binding count but improves readability.
- Test functions with sequential setup steps, where the locals are test fixtures.
- Builder-pattern chains that are bound to locals for readability (e.g., `let client = ClientBuilder::new()...`).
- Functions where the locals are genuinely independent and extraction would add indirection without clarity.

**Related rules:** N32 `excessive-function-length`, N40 `multiple-responsibilities`.
