# Rule: missing-must-use

**Severity:** suggestion

Inspired by [Power of 10 Rule 7](https://en.wikipedia.org/wiki/The_Power_of_10:_Rules_for_Developing_Safety-Critical_Code): *"Each calling function must check the return value of non-void functions."*

Rust's `Result` and `Option` types already carry `#[must_use]`, so ignoring a `Result` produces a compiler warning by default. This rule addresses the gap: functions and types that return important values but are not marked `#[must_use]`, making it easy to silently ignore their output.

## Problem

A public function returns a value that is meaningless to discard (e.g., a builder, a guard, an error indicator), but callers can ignore the return value with no compiler warning. This leads to bugs where callers assume the function mutates state in-place but it actually returns a new value, or where a critical resource guard is immediately dropped.

## Example

### Bad
```rust
pub struct Query { /* ... */ }

impl Query {
    /// Adds a filter clause.
    pub fn filter(mut self, clause: &str) -> Self {
        self.clauses.push(clause.to_string());
        self
    }
}

// Caller bug — the filtered query is silently discarded
let q = Query::new();
q.filter("active = true");  // Returns a new Query, original is unchanged
```

### Good
```rust
#[must_use = "this returns a new Query with the filter applied; the original is unchanged"]
pub struct Query { /* ... */ }

impl Query {
    pub fn filter(mut self, clause: &str) -> Self {
        self.clauses.push(clause.to_string());
        self
    }
}

// Now the compiler warns if the return value is unused
```

### Also consider for functions
```rust
// Bad — caller might forget to check the returned bool
pub fn remove_entry(map: &mut HashMap<K, V>, key: &K) -> bool { /* ... */ }

// Good
#[must_use = "returns whether the entry was present"]
pub fn remove_entry(map: &mut HashMap<K, V>, key: &K) -> bool { /* ... */ }
```

## When to flag

- Builder types or builder methods that return `Self` — discarding the return value is almost always a bug.
- Functions that return a guard or handle (e.g., `MutexGuard`, file lock) that serves a purpose only while held.
- Pure functions whose only purpose is to compute and return a value — calling them for side effects is meaningless.
- Functions returning a `bool` or status code indicating success/failure that callers should check.

## When NOT to flag

- Functions returning `Result` or `Option` — these are already `#[must_use]` by default.
- Functions where ignoring the return value is a common and intentional pattern (e.g., `Vec::push` returns `()`, `HashMap::insert` returns the old value which is often not needed).
- Private functions where all call sites in the crate already use the return value.
- Functions returning `()`.
