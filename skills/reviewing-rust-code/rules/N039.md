# Rule: non-inclusive-terminology

**Severity:** warning

Inspired by [Linux kernel coding style §4](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#naming): the kernel guide explicitly adopted inclusive terminology replacements for identifiers and documentation.

## Problem

Code uses terminology that has been superseded by more inclusive alternatives. The technical meaning is preserved by the replacements; the only change is that the metaphor is neutral. Using outdated terms can make the codebase less welcoming and conflict with organizational contribution policies.

## Example

### Bad
```rust
struct ReplicationConfig {
    master_address: SocketAddr,
    slave_addresses: Vec<SocketAddr>,
}

fn check_blacklist(domain: &str, blacklist: &[String]) -> bool {
    blacklist.contains(&domain.to_string())
}
```

### Good
```rust
struct ReplicationConfig {
    primary_address: SocketAddr,
    secondary_addresses: Vec<SocketAddr>,
}

fn check_denylist(domain: &str, denylist: &[String]) -> bool {
    denylist.contains(&domain.to_string())
}
```

## When to flag

- `master` / `slave` in identifiers, type names, or doc comments — use `primary` / `secondary`, `leader` / `follower`, or `controller` / `target` instead.
- `whitelist` / `blacklist` in identifiers, type names, or doc comments — use `allowlist` / `denylist` or `passlist` / `blocklist` instead.

## When NOT to flag

- References to external protocols, APIs, or standards that use legacy terminology (e.g., a field matching a wire protocol name). In these cases, use the external term at the boundary and map to inclusive terms internally.
- Strings or comments that quote external documentation verbatim.
- Historical commit messages or changelogs.
