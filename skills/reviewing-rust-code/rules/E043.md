# Rule: manual-result-forwarding

**Severity:** warning

Inspired by [Linux kernel coding style §7](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#centralized-exiting-of-functions): the kernel guide recommends `goto` labels for centralized cleanup. In Rust, the `?` operator and `Drop` serve the same purpose — they ensure cleanup runs and errors propagate without scattering handling logic across every call site.

## Problem

A function manually matches on `Result` or `Option` to propagate errors upward, when the `?` operator would produce the same behavior with less noise. Deeply nested `match` arms obscure the happy path and increase the chance of forgetting an error case. The `?` operator is Rust's idiomatic mechanism for early-return error propagation; reimplementing it manually adds boilerplate.

## Example

### Bad
```rust
fn load_and_validate(path: &Path) -> Result<Config> {
    match fs::read_to_string(path) {
        Ok(content) => match parse_config(&content) {
            Ok(config) => match validate(&config) {
                Ok(()) => Ok(config),
                Err(e) => Err(e.into()),
            },
            Err(e) => Err(e.into()),
        },
        Err(e) => Err(e.into()),
    }
}
```

### Good
```rust
fn load_and_validate(path: &Path) -> Result<Config> {
    let content = fs::read_to_string(path)?;
    let config = parse_config(&content)?;
    validate(&config)?;
    Ok(config)
}
```

Adding context via `map_err` or `.context()` before `?` is encouraged — this is not a violation:
```rust
fn load_and_validate(path: &Path) -> Result<Config> {
    let content = fs::read_to_string(path)
        .with_context(|| format!("failed to read {}", path.display()))?;
    let config = parse_config(&content)
        .context("invalid config format")?;
    validate(&config)?;
    Ok(config)
}
```

## When to flag

- `match` on `Result` where every `Err` arm simply wraps and returns the error (i.e., reimplements `?`).
- Nested `match` on `Result`/`Option` that creates a "staircase" pattern, when `?` would flatten it.
- `if let Ok(x) = expr { ... } else { return Err(...) }` patterns that are equivalent to `let x = expr?;`.

## When NOT to flag

- `match` on `Result` where different error variants are handled differently (e.g., retry on timeout, fail on permission denied).
- `match` that adds context, transforms the error type, or performs recovery — this is genuine error handling, not forwarding.
- Cases where `?` cannot be used because the function's return type does not implement `From` for the error type and adding the conversion is undesirable.
- `match` on `Result` inside closures where `?` would return from the closure, not the enclosing function.
