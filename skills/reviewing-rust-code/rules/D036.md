# D036: suppressed-warning

**Level:** default

## Problem

A `#[allow(...)]` attribute suppresses a compiler or Clippy warning instead of fixing the underlying issue.
Suppressed warnings hide real problems: dead code that indicates a missing feature, unused variables that reveal a logic error, or Clippy lints that catch common mistakes.
Over time, suppressed warnings accumulate and erode the value of the warning system.

## Pros / Cons / Decision

**Pros of addressing warnings instead of suppressing them:**
- Warnings catch real issues early.
- Keeps the warning system trustworthy.

**Cons:**
- Some warnings are false positives in specific contexts.
- Generated code may trigger warnings that cannot be fixed at the source.

**Decision:**
Fix the underlying issue instead of suppressing the warning.
When suppression is genuinely necessary,
scope it narrowly and add a comment explaining why.

## Example

### Bad
```rust
#[allow(unused_variables)]
fn process(data: &[u8], config: &Config) -> Result<()> {
    // `config` is never used — was it forgotten, or is it intentional?
    parse(data)
}

#[allow(clippy::needless_return)]
fn calculate(x: f64) -> f64 {
    // Just remove the explicit return instead of suppressing the lint
    return x * 2.0;
}
```

### Good
```rust
fn process(data: &[u8], _config: &Config) -> Result<()> {
    // Prefixing with _ indicates intentionally unused
    parse(data)
}

fn calculate(x: f64) -> f64 {
    x * 2.0
}
```

When suppression is genuinely necessary, scope it narrowly and explain why:
```rust
// This triggers `clippy::cast_possible_truncation` but the value is
// guaranteed ≤ 255 by the protocol spec (§4.2).
#[allow(clippy::cast_possible_truncation)]
let flags = raw_flags as u8;
```

## When to flag

- `#[allow(dead_code)]` in non-prototype code — dead code should be removed.
- `#[allow(unused_variables)]` or `#[allow(unused_imports)]` — prefix with `_` or remove instead.
- `#[allow(...)]` without a comment explaining why the suppression is necessary.
- Blanket `#![allow(...)]` at the crate root that suppresses warnings project-wide, especially `clippy::all` or `warnings`.
- `#[allow(clippy::...)]` for a lint that could be fixed with a simple code change.

## When NOT to flag

- `#[allow(...)]` with an accompanying comment explaining why the lint is a false positive or inapplicable.
- `#[allow(unused)]` in early prototyping or WIP code (though it should be removed before merge).
- `#[allow(...)]` on generated code or macro expansions where the lint cannot be addressed in source.
- `#[allow(clippy::too_many_arguments)]` or similar where refactoring would reduce clarity, provided a justification is given.
- `#[cfg(test)]` modules where `#[allow(unused)]` is common for test helpers.

[^1]: Inspired by [Power of 10 Rule 10](https://en.wikipedia.org/wiki/The_Power_of_10:_Rules_for_Developing_Safety-Critical_Code): *"Compile with all possible warnings active; all warnings should then be addressed before release of the software."*
