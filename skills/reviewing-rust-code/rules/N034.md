# Rule: complex-macro

**Severity:** warning

Inspired by [Power of 10 Rule 8](https://en.wikipedia.org/wiki/The_Power_of_10:_Rules_for_Developing_Safety-Critical_Code): *"The use of the preprocessor must be limited to the inclusion of header files and simple macro definitions."*

Also inspired by [Linux kernel coding style §12](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#macros-enums-and-rtl): the kernel guide warns against macros that "affect control flow" and macros that "resemble functions" but bypass type checking. In Rust, the `?` operator is the sanctioned way to do early-return error propagation; there is no reason to reinvent it in a macro.

## Problem

A `macro_rules!` or proc macro is complex enough that it is difficult to read, produces opaque compiler errors for users, or defies static analysis tools. Macros bypass normal Rust tooling — `rust-analyzer` cannot always expand them for go-to-definition, `cargo clippy` may not lint their output, and compile errors point at the call site rather than the bug.

## Example

### Bad
```rust
macro_rules! define_handlers {
    ($($method:ident => $path:expr => $handler:ident),* $(,)?) => {
        $(
            macro_rules! $handler {
                ($req:expr) => {{
                    // Nested macro definition — very hard to follow
                    let resp = process($req, stringify!($method));
                    validate_response(resp, $path)
                }};
            }
        )*
    };
}
```

### Good
```rust
// Use a function and a simple data structure instead
struct Route {
    method: Method,
    path: &'static str,
    handler: fn(Request) -> Response,
}

const ROUTES: &[Route] = &[
    Route { method: Method::GET,  path: "/users", handler: get_users },
    Route { method: Method::POST, path: "/users", handler: create_user },
];
```

Or, if a macro is warranted, keep it simple:
```rust
macro_rules! route {
    ($method:ident, $path:expr, $handler:expr) => {
        Route {
            method: Method::$method,
            path: $path,
            handler: $handler,
        }
    };
}
```

## When to flag

- `macro_rules!` with nested macro definitions.
- Macros that contain `return`, `break`, `continue`, or `?` that affect control flow in the calling function — a reader of the call site must be able to understand the control flow without expanding the macro.
- Macros that generate `impl` blocks with complex trait logic — consider a derive macro or a generic function instead.
- Macros used where a generic function, trait, or const generic would achieve the same goal with better tooling support.
- Macros that accept complex syntax that is not obviously Rust (DSLs that surprise readers unfamiliar with the macro).

## When NOT to flag

- Simple declarative macros that reduce clear boilerplate (e.g., `vec![]`-style constructors, repetitive test case generation).
- Derive macros from well-known crates (`serde`, `clap`, `thiserror`).
- Macros that exist specifically because Rust's type system cannot express the pattern (e.g., variadic arguments, compile-time string formatting).
- Macros in test code for generating test fixtures.
