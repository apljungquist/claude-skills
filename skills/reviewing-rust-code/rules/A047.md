# Rule: undocumented-shared-ownership

**Severity:** warning

Inspired by [Linux kernel coding style §11](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#data-structures): *"If another thread can find your data structure, and you don't have a reference count on it, you almost certainly have a bug."* The kernel requires reference counting for structures visible outside single-threaded contexts. In Rust, `Arc`/`Rc` serve this role, but overusing them hides ownership relationships and can create reference cycles that leak memory.

## Problem

`Arc` or `Rc` is used without a comment explaining why shared ownership is necessary. Each `Arc` is a statement that "multiple parts of the system need to independently own this value." If that statement is false, the `Arc` adds unnecessary complexity, runtime overhead (atomic reference counting), and obscures which component is responsible for the value's lifecycle.

## Example

### Bad
```rust
// Why is this Arc? Is sharing actually needed?
let config: Arc<Config> = Arc::new(load_config()?);
// If only one component uses config, Box or a reference suffices.

// Everything is Arc<Mutex<T>> — nobody knows who owns what
let state = Arc::new(Mutex::new(AppState::default()));
let db = Arc::new(Mutex::new(Database::connect()?));
let cache = Arc::new(Mutex::new(HashMap::new()));
```

### Good
```rust
/// Shared between the request handler and the background metrics
/// collector, both of which outlive individual requests.
let stats: Arc<Stats> = Arc::new(Stats::default());

// Only used by the main thread — no Arc needed
let config = load_config()?;
```

## When to flag

- `Arc::new(...)` or `Rc::new(...)` without a nearby comment explaining which components share the value and why independent ownership is required.
- `Arc<Mutex<T>>` used pervasively across a codebase without documentation of the sharing pattern — suggests the code is defaulting to shared ownership instead of designing ownership.
- `Arc` used for a value that is only ever cloned once or never cloned at all.

## When NOT to flag

- `Arc` in well-understood patterns like `Arc<dyn Trait>` for trait objects shared across threads, where the sharing need is obvious from context.
- `Arc` required by framework APIs (e.g., `tokio::spawn` requires `'static`, which often necessitates `Arc`).
- `Rc` in single-threaded code with clear tree/graph data structures where shared references are inherent to the structure.
- A comment, doc comment, or type alias (e.g., `type SharedState = Arc<Mutex<State>>`) already explains the sharing intent.
