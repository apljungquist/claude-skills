# Rule: type-alias-over-newtype

**Severity:** suggestion

Inspired by [Linux kernel coding style §5](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#typedefs): the kernel guide warns against typedefs that hide the actual type without adding safety. *"It's a mistake to use typedef for structures and pointers."* In Rust, `type` aliases are transparent to the compiler — `type UserId = u64` and `type OrderId = u64` are interchangeable, so you can pass an `OrderId` where a `UserId` is expected without a compiler error.

## Problem

A `type` alias is used where two values have the same underlying type but different semantics. The alias provides documentation value but no compile-time safety — the compiler treats the alias and the underlying type as identical. A newtype wrapper (`struct Meters(f64)`) catches type confusion at compile time.

## Example

### Bad
```rust
type UserId = u64;
type OrderId = u64;

fn get_user_orders(user_id: UserId) -> Vec<OrderId> {
    // ...
}

// Compiles without error — silent bug:
let order_id: OrderId = 42;
let orders = get_user_orders(order_id); // Passes an OrderId as a UserId
```

### Good
```rust
struct UserId(u64);
struct OrderId(u64);

fn get_user_orders(user_id: UserId) -> Vec<OrderId> {
    // ...
}

// Compiler error: expected `UserId`, found `OrderId`
let order_id = OrderId(42);
let orders = get_user_orders(order_id); // Does not compile
```

## When to flag

- `type Alias = Primitive` where another alias over the same primitive exists in the codebase and the two represent semantically different concepts (e.g., `type UserId = u64` alongside `type OrderId = u64`).
- `type` alias for a domain concept that has invariants (e.g., `type PortNumber = u16` — not all `u16` values are valid ports).
- Multiple function parameters with the same underlying type that represent different domain concepts, using type aliases for clarity but not safety.

## When NOT to flag

- `type` aliases that shorten complex generic types without adding semantic meaning (e.g., `type Result<T> = std::result::Result<T, MyError>`).
- `type` aliases used in trait definitions as associated types.
- Cases where the newtype boilerplate (implementing `Display`, `From`, `Serialize`, etc.) would outweigh the safety benefit and the type confusion risk is low.
- `type` aliases for callback signatures or closure types (e.g., `type Handler = Box<dyn Fn(Request) -> Response>`).
