# Rule: excessive-function-length

**Severity:** suggestion

Inspired by [Power of 10 Rule 4](https://en.wikipedia.org/wiki/The_Power_of_10:_Rules_for_Developing_Safety-Critical_Code): *"No function should be longer than what can be printed on a single sheet of paper."*

Also inspired by [Linux kernel coding style §6](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#functions): *"Functions should be short and sweet, and do just one thing. They should fit on one or two screenfuls of text."* The kernel guide also recommends limiting local variables to 5–10, because "the human brain can generally easily keep track of about 7 different things" (see N41 `too-many-locals`).

## Problem

A function is long enough that a reviewer cannot see the entire body at once, making it difficult to reason about control flow, verify correctness, and maintain the code. Long functions tend to accumulate multiple responsibilities and become harder to test in isolation.

## Example

### Bad
```rust
fn process_order(order: &Order, db: &Database) -> Result<Receipt> {
    // 150+ lines handling validation, pricing, tax calculation,
    // inventory check, payment processing, email notification,
    // and audit logging all in one function.
    // ...
}
```

### Good
```rust
fn process_order(order: &Order, db: &Database) -> Result<Receipt> {
    validate_order(order)?;
    let pricing = calculate_pricing(order)?;
    let tax = calculate_tax(&pricing, order.region())?;
    check_inventory(db, order)?;
    let payment = charge_payment(order, pricing.total + tax)?;
    send_confirmation(order, &payment)?;
    record_audit_log(db, order, &payment)?;
    Ok(Receipt::new(order, payment, tax))
}
```

## When to flag

- A function exceeds roughly 60 lines of executable code (excluding blank lines, comments, and closing braces). This is a guideline, not a hard cutoff — use judgment.
- A function has multiple clearly separable responsibilities that could be extracted.
- A function requires scrolling to see both the setup and the return value.

## When NOT to flag

- A function is long primarily due to a large `match` with many arms, where each arm is simple and extracting them would reduce clarity.
- Declarative builders or configuration code where the length is inherent to the data (e.g., a long chain of `.field(...)` calls).
- Test functions with sequential setup-execute-assert steps that would lose clarity if split.
- Generated code.
