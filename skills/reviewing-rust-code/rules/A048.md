# Rule: reimplemented-std-facility

**Severity:** warning

Inspired by [Linux kernel coding style §18](https://www.kernel.org/doc/html/v6.4/process/coding-style.html#don-t-re-invent-the-kernel-macros): *"The kernel provides a number of macros and inline functions. Please don't reinvent them."* The same applies to Rust's standard library and well-established ecosystem crates. A hand-rolled utility is unlikely to be as correct, as fast, or as well-tested as the standard version.

## Problem

Code reimplements functionality that already exists in the standard library or in a well-known crate the project already depends on. Every custom reimplementation is code that must be maintained, documented, and tested. The failure mode is a subtle bug in a hand-rolled utility that would never have occurred with the standard version — discovered in production, after months of use.

## Example

### Bad
```rust
// Hand-rolled word counting with a Vec of tuples instead of HashMap
let mut counts: Vec<(&str, usize)> = Vec::new();
for word in words {
    if let Some(entry) = counts.iter_mut().find(|(w, _)| *w == word) {
        entry.1 += 1;
    } else {
        counts.push((word, 1));
    }
}

// Hand-rolled min/max instead of Iterator::min/max
let mut min_val = values[0];
for &v in &values[1..] {
    if v < min_val {
        min_val = v;
    }
}
```

### Good
```rust
use std::collections::HashMap;

let mut counts: HashMap<&str, usize> = HashMap::new();
for word in words {
    *counts.entry(word).or_insert(0) += 1;
}

let min_val = values.iter().copied().min().expect("non-empty slice");
```

## When to flag

- Manual iteration to find min/max/sum when `Iterator` methods exist.
- Custom collection types that replicate `HashMap`, `BTreeMap`, `VecDeque`, etc.
- Hand-rolled string parsing for formats handled by `str::parse`, `FromStr`, or well-known crates like `serde`.
- Manual mutex-based channels when `std::sync::mpsc` or `tokio::sync` channels exist.
- Custom retry/backoff logic when the project already depends on a retry crate.
- Re-implementing `Default`, `Display`, or other standard trait functionality without using the trait.

## When NOT to flag

- Performance-critical paths where benchmarking demonstrates the standard facility is a bottleneck, with benchmark results documented in a comment.
- `#[no_std]` environments where the standard library facility is not available.
- Cases where the standard facility does not quite fit and the custom version has meaningfully different semantics (not just a wrapper).
- Prototyping or exploratory code where reaching for the right crate would slow down iteration.
